## 트리거 조건

다음 상황에서 **반드시** 이 스킬을 사용하십시오 (직접 디버깅 금지):
- 에러, 500 에러, 스택 트레이스 보고
- 예상치 못한 동작, "어제까지 동작했음"
- 무언가가 중단된 이유 파악 필요

---

## 4단계 조사 프로세스

### Phase 1: 조사 (Investigate)

```bash
# 1. 에러 로그 수집 (skills/logql-query.md 참조)
# 에러 발생 시간대, traceId, userId 특정

# 2. 재현 시도
# - 동일 환경에서 단계별 재현
# - 재현 가능하면 영상 녹화 시작 (skills/video-capture.md)

# 3. 범위 파악
# - 언제부터 발생했는가? (git log --since, 배포 이력)
# - 어떤 사용자/환경에서 발생하는가?
# - 얼마나 자주 발생하는가?
git log --oneline --since="2 days ago" -- [관련 파일]
```

### Phase 2: 분석 (Analyze)

```bash
# 관련 코드 경로 추적
grep -r "[에러 메시지 키워드]" src/ --include="*.ts" --include="*.java" --include="*.py"

# 최근 변경사항 확인
git diff HEAD~5 -- [의심 파일]

# 의존성 체인 추적
# 에러 발생 지점에서 호출 스택을 역추적
```

### Phase 3: 가설 (Hypothesize)

근본 원인 가설을 명시적으로 작성하십시오:

```
가설 1: [구체적 원인] — 신뢰도: [높음/중간/낮음]
  근거: [증거]
  검증 방법: [어떻게 확인할 것인가]

가설 2: [구체적 원인] — 신뢰도: [높음/중간/낮음]
  근거: [증거]
  검증 방법: [어떻게 확인할 것인가]
```

**가장 신뢰도 높은 가설부터 검증하십시오.**

### Phase 4: 구현 (Implement)

근본 원인이 확인된 후에만 수정을 시작하십시오:

1. 근본 원인 문서화 (PR 본문에 반드시 포함)
2. 최소한의 수정 구현 (증상이 아닌 원인 수정)
3. 재현 시나리오로 수정 검증
4. 회귀 테스트 추가 (동일 버그 재발 방지)
5. `skills/pr-lifecycle.md` 절차로 PR 오픈

## 레드팀 자가 검증

수정 전 다음을 스스로 물어보십시오:
- 정말 근본 원인인가, 아니면 증상인가?
- 이 수정이 다른 곳을 깨뜨릴 수 있는가?
- 동시성 시나리오에서 이 수정이 유지되는가?
- 이 버그의 다른 발현 형태가 있는가?

## 에스컬레이션 기준

4단계를 완료했지만 근본 원인을 찾지 못하면:
1. `SIGNALS.md`에 실패 로그 추가 (조사 결과, 시도한 가설 포함)
2. 사람에게 에스컬레이션 (무엇을 시도했는지, 무엇이 의심스러운지 명시)
3. 추측 기반 수정 절대 금지
