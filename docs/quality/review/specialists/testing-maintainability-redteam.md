# 테스팅 스페셜리스트 리뷰 체크리스트
> 트리거: 항상 (모든 리뷰)

---

## 음성 경로 테스트 누락
- 에러, 거부, 잘못된 입력을 처리하는 새 코드 경로에 대응 테스트 없음
- 테스트 없는 가드 절 및 조기 반환
- try/catch, rescue, 에러 경계의 에러 분기에 실패 경로 테스트 없음
- 코드에서 assert되지만 "거부됨" 케이스를 테스트하지 않는 권한/인증 확인

## 엣지 케이스 커버리지 누락
- 경계값: 0, 음수, max-int, 빈 문자열, 빈 배열, null/undefined
- 단일 요소 컬렉션 (루프의 off-by-one)
- 사용자 대면 입력의 유니코드 및 특수 문자
- 경쟁 조건 테스트 없는 동시 접근 패턴

## 테스트 격리 위반
- 가변 상태를 공유하는 테스트 (클래스 변수, 전역 싱글톤, 정리 안 된 DB 레코드)
- 순서 의존적 테스트 (순서대로 실행 시 통과, 랜덤화 시 실패)
- 시스템 시계, 시간대, 로케일에 의존하는 테스트
- stub/mock 대신 실제 네트워크 호출을 하는 테스트

## 플레이키 테스트 패턴
- 타이밍 의존적 assertions (sleep, setTimeout, 타이트한 타임아웃의 waitFor)
- 순서 없는 결과에 대한 순서 assertions (해시 키, Set 순회)
- 폴백 없이 외부 서비스에 의존하는 테스트
- 시드 제어 없는 랜덤 테스트 데이터

## 보안 강제 테스트 누락
- "미인가" 케이스 테스트 없는 컨트롤러의 인증/인가 확인
- 실제로 차단되는지 증명하는 테스트 없는 속도 제한 로직
- 악의적 입력 테스트 없는 입력 sanitization
- 통합 테스트 없는 CSRF/CORS 설정

# 유지보수성 스페셜리스트 리뷰 체크리스트
> 트리거: 항상 (모든 리뷰)

---

## 죽은 코드 & 미사용 import
- 변경된 파일에서 할당 후 읽지 않는 변수
- 정의되었지만 호출되지 않는 함수/메서드 (repo 전체 Grep 확인)
- 변경 후 더 이상 참조되지 않는 import/require
- 주석 처리된 코드 블록 (제거하거나 이유를 설명하십시오)

## 매직 넘버 & 문자열 결합
- 로직에 사용된 베어 숫자 리터럴 (임계값, 제한, 재시도 횟수) → 명명된 상수로
- 다른 곳의 쿼리 필터 또는 조건으로 사용되는 에러 메시지 문자열
- 설정이어야 할 하드코딩된 URL, 포트, 호스트명
- 여러 파일에 걸쳐 중복된 리터럴 값

## stale 주석 & 문서
- 이 diff에서 코드가 변경된 후 이전 동작을 설명하는 주석
- 완료된 작업을 참조하는 TODO/FIXME 주석
- 현재 함수 시그니처와 일치하지 않는 파라미터 목록의 docstring

## DRY 위반
- diff 내 여러 번 나타나는 유사 코드 블록 (3줄 이상)
- 공유 헬퍼가 더 깔끔한 복사-붙여넣기 패턴
- 테스트 파일 간 중복된 설정 로직

## 조건부 사이드 이펙트
- 한 분기에서 관련 레코드를 업데이트하지만 다른 분기는 업데이트하지 않는 상태 전환
- 액션이 조건부로 건너뛰어졌는데 발생했다고 주장하는 로그 메시지
- 행복 경로에서만 발생하고 에러/엣지 경로를 놓치는 이벤트 발송

## 모듈 경계 위반
- 다른 모듈의 내부 구현 접근 (관례상-private 메서드)
- service/model을 통해야 하는 컨트롤러/뷰의 직접 DB 쿼리
- 인터페이스를 통해 통신해야 하는 컴포넌트 간 강한 결합

# 레드팀 리뷰 가이드
> 트리거: diff > 200 라인 OR 보안 스페셜리스트가 CRITICAL 발견 시
> 이것은 체크리스트 리뷰가 아닙니다. 적대적 분석입니다.
> 다른 스페셜리스트가 놓친 것을 찾으십시오.

---

## 접근법

### 1. 행복 경로를 공격하라
- 정상 부하의 10배에서 어떻게 되는가?
- 두 요청이 동시에 같은 리소스에 접근하면 어떻게 되는가?
- DB가 느릴 때 (>5초 쿼리 시간) 어떻게 되는가?
- 외부 서비스가 쓰레기를 반환하면 어떻게 되는가?

### 2. 조용한 실패를 찾아라
- 예외를 삼키는 에러 처리 (로그만 있는 catch-all)
- 부분적으로 완료될 수 있는 작업 (5개 중 3개 처리 후 충돌)
- 실패 시 일관성 없는 상태로 레코드를 남기는 상태 전환
- 아무에게도 알리지 않고 실패하는 백그라운드 잡

### 3. 신뢰 가정을 악용하라
- 프론트엔드에서 검증되지만 백엔드에서는 검증되지 않는 데이터
- 인증 없이 호출되는 내부 API ("우리 코드만 이걸 호출한다"고 가정)
- 존재한다고 가정하지만 검증하지 않는 설정 값
- sanitization 없이 사용자 입력으로 구성된 파일 경로 또는 URL

### 4. 엣지 케이스를 깨뜨려라
- 최대 가능 입력 크기에서 어떻게 되는가?
- 0개 항목, 빈 문자열, null 값에서 어떻게 되는가?
- 첫 실행(기존 데이터 없음)에서 어떻게 되는가?
- 사용자가 100ms 내 버튼을 두 번 클릭하면 어떻게 되는가?

### 5. 다른 스페셜리스트가 놓친 것을 찾아라
- 각 스페셜리스트의 발견을 검토하십시오. 카테고리 간 갭이 무엇인가?
- 교차 카테고리 이슈 (성능 이슈이기도 한 보안 이슈)
- 통합 경계의 이슈 (두 시스템이 만나는 곳)
- 특정 배포 설정에서만 나타나는 이슈
