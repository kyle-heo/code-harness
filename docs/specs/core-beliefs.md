# 핵심 철학 (Core Beliefs)

> 이 파일은 제품과 엔지니어링 의사결정의 북극성입니다.
> 트레이드오프 상황에서 이 원칙을 기준으로 판단하십시오.

---

## 제품 철학

1. **사용자 문제 우선**: 기술적으로 흥미롭더라도 사용자 문제를 해결하지 않으면 구현하지 않는다.
2. **단순함 우선**: 복잡한 해결책보다 단순한 해결책을 선호한다. 단순함은 나중에 복잡하게 만들 수 있지만, 복잡함은 되돌리기 어렵다.
3. **빠른 피드백 루프**: 사용자에게 빠르게 도달하는 것이 완벽한 것보다 중요하다.

---

## 엔지니어링 철학

1. **수동 코드 없음**: 모든 코드는 에이전트가 작성한다. 이 제약이 도구와 환경 투자를 강제한다.
2. **실패는 신호다**: 에이전트가 실패하면 원인을 구조화하고 불변 조건에 인코딩한다. "더 분발"은 해결책이 아니다.
3. **제약이 속도를 만든다**: 린터, 레이어 규칙, 불변 조건은 에이전트가 빠르게 움직일 수 있는 레일이다. 없으면 드리프트가 발생한다.
4. **컨텍스트는 리포지터리 안에**: 에이전트가 접근할 수 없는 지식(Slack, Google Docs, 사람 머릿속)은 존재하지 않는 것과 같다.

---

## 의사결정 프레임워크

트레이드오프 상황에서 다음 순서로 우선순위를 부여한다:

```
1. 사용자 안전 및 데이터 보안
2. 시스템 신뢰성 (SLO 준수)
3. 코드베이스 일관성 (CONSTRAINTS.md 준수)
4. 개발 속도
5. 기술적 우아함
```
