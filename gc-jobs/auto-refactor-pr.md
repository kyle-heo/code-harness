# gc-jobs/auto-refactor-pr.md
> 품질 점수 하락 기여 파일을 자동으로 리팩터링하고 PR을 생성합니다.
> 트리거: quality-grader가 점수 < 80 판단 시

---

## 실행 원칙

- 한 번에 최대 3개 파일만 리팩터링 (범위 과대 방지)
- 기능 변경 없이 구조 개선만 수행 (테스트가 동일하게 통과해야 함)
- 리팩터링 전후 테스트 실행 필수

---

## 실행 지시사항 (Claude Code에게)

### 1단계: 대상 파일 선정

`quality-grader.md` 결과에서 점수 하락 기여 상위 3개 파일 선정.
선정 기준 (우선순위 순):
1. C-001 레이어 위반 파일 (아키텍처 정합성)
2. C-002 경계 검증 누락 파일 (보안)
3. 파일 크기 초과 파일 (유지보수성)
4. 공유 유틸리티 미활용 파일 (중복)

### 2단계: 리팩터링 전 안전 확인

```bash
# 변경 전 테스트 전체 통과 확인
npx jest --testPathPattern=[대상파일] --passWithNoTests

# 변경 전 스냅샷 (UI 컴포넌트인 경우)
# skills/chrome-devtools.md 참조
```

### 3단계: 리팩터링 수행

CONSTRAINTS.md 규칙 준수 하에 다음 순서로 수행:
1. 레이어 위반 import 수정
2. 경계 데이터 파싱 추가 (Zod 스키마)
3. 과대 파일 분리 (레이어별 파일로 분할)
4. 공유 유틸리티로 중복 코드 교체

### 4단계: 리팩터링 후 검증

```bash
# 기능 동일성 확인
npx jest --testPathPattern=[대상파일]

# 린터 통과 확인
node linters/layer-deps.js [대상파일]
node linters/boundary-check.js [대상파일]
```

### 5단계: PR 오픈

```
제목: refactor: [파일명] 품질 개선 (C-[번호] 준수)
본문:
- 변경 이유: quality-grader 트리거 (점수: X -> 예상 Y)
- 변경 내용: [구체적 변경 목록]
- 테스트: 전후 동일 통과 확인
- 린터: 전체 통과 확인
```

agent-to-agent 리뷰 후 자동 병합.
1분 내 검토 가능한 크기로 PR 유지.

## 실행 주기

- quality-grader 트리거 시
- 전체 품질 점수 < 80 감지 시 (대상: 상위 3파일)
- 전체 품질 점수 < 70 감지 시 (대상: 상위 3파일 + 사람 에스컬레이션)
